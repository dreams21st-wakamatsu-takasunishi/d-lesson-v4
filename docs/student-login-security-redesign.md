# 児童ログインと情報漏洩防止の再設計

## 優先順位

1. 児童名、進捗、コイン、練習履歴などの情報漏洩を防ぐ。
2. 児童がデスクトップのURLアイコンから開き、自分の練習を始めやすくする。

この2つを両立するため、公開URLでは「ログイン前に児童名一覧を出さない」ことを基本方針にします。

## 結論

本番運用では、児童ごとにSupabase Authアカウントを作ります。

ただし、児童にUUIDやメールアドレスを意識させません。児童が使う画面では、次のような短い入力に置き換えます。

```text
児童番号: 12
あいことば: ひまわり49
```

内部的には、この入力からDレッスン用のログインメールアドレスを組み立て、Supabase Authでログインします。ログイン後、RLSにより、その児童本人の `user_data` だけが読み書きできます。

## なぜ児童名一覧をログイン前に出さないか

「自分の名前を選ぶ」方式は使いやすい一方、公開URLでは次のリスクがあります。

- 未ログインの人に児童名一覧が見える。
- 画面共有、スクリーンショット、ブラウザキャッシュから名簿が見える。
- URLを知った第三者がアクセスしたとき、児童が在籍していることが分かる。
- 児童名をクリックするだけで進める設計にすると、なりすましを防ぎにくい。

そのため、本番ではログイン前に出す情報を「学校名やDレッスン名」程度に抑え、児童名はログイン後に初めて表示します。

## 推奨ログイン体験

### 1. 児童がデスクトップアイコンから開く

デスクトップのショートカットは公開URLを開きます。

```text
https://dreams21st-wakamatsu-takasunishi.github.io/d-lesson-v4/
```

必要なら将来、教室や曜日をURLに含めます。ただし、これは秘密情報にはしません。

```text
https://dreams21st-wakamatsu-takasunishi.github.io/d-lesson-v4/?room=monday-a
```

### 2. 児童ログイン画面を表示

ログイン前は名簿を出しません。

表示する入力:

```text
児童番号
あいことば
```

低学年向けには、児童番号をボタンで選び、あいことばだけ入力する画面にできます。ただし番号ボタンに児童名は出しません。

### 3. ログイン成功後に自動で本人データへ入る

ログイン成功後、`lesson_user_access` を読みます。

児童ロールで許可されている `user_data_id` が1件だけなら、その児童を自動選択します。

```text
ログイン成功
↓
RLSで本人の user_data だけ取得
↓
「ようこそ、○○さん」
↓
練習メニュー
```

### 4. 児童名はログイン後だけ表示

本人確認のため、ログイン後にだけ名前を表示します。

```text
ようこそ、山田 太郎さん
```

これにより、公開URLを開いただけでは名簿が漏れません。

## アカウント設計

### 児童

児童ごとに1つのSupabase Authアカウントを作ります。

例:

```text
画面で使う児童番号: 12
内部ログインメール: dlesson-student-012@d-lesson.local
Auth User ID: 36c95d17-621a-447d-b44b-638e21219ca1
user_data_id: student_xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
```

児童本人は、メールアドレスやAuth User IDを見ません。

### 先生

先生アカウントは、先生ごと、または教室運用単位で作ります。

先生は読み取り中心です。児童の練習進捗を書き換えない設計を維持します。

### 管理者

管理者アカウントは少数に限定します。

管理者だけが児童追加、バックアップ復元、Auth紐づけ、設定変更を行えます。

## パスワード、あいことばの考え方

公開URLでは4桁数字だけのPINは弱いです。

推奨は、児童が読める短い合言葉と数字を組み合わせる方式です。

```text
ひまわり49
さくら72
みどり18
```

より安全にするなら、次のようにします。

- 児童番号とあいことばを別々にする。
- あいことばは児童ごとに異なる。
- 年度や学期ごとに更新する。
- 紙カードで配布し、画面には表示しない。
- 失敗が多いアカウントは管理者がパスワード変更する。

## セッション設計

共有PCでは、ログイン状態が残ること自体がリスクになります。

本番では次を目指します。

- 児童ログインはブラウザを閉じたら切れる。
- 一定時間操作がなければ自動ログアウトする。
- トップ画面にログアウトボタンを常に表示する。
- 先生、管理者は必要に応じてログイン保持を検討する。

実装上は、Supabase Authの保存先を児童向けには `sessionStorage` 寄りにするか、自動ログアウトを強めます。

## RLS設計

現在の方針を維持します。

```text
lesson_user_access.auth_user_id
  = Supabase Auth User ID

lesson_user_access.user_data_id
  = student_... の児童データID

lesson_user_access.role
  = student / teacher / admin
```

児童:

```text
自分の user_data_id だけ読み書き可能
```

先生:

```text
許可されたグループの児童だけ読み取り可能
進捗保存は不可
```

管理者:

```text
全児童、設定、Auth連携を管理可能
```

## 非推奨案

### 共有の教室アカウントで名簿から選ぶ

使いやすいですが、情報漏洩防止を最優先するなら推奨しません。

理由:

- 共有アカウントに複数児童の閲覧権限を与える必要がある。
- そのログイン状態が残ると、別の児童が他児童のデータを見られる。
- トークンが漏れた場合、範囲内の全児童データが読まれる。

どうしても低学年運用で必要な場合のみ、将来「教室端末モード」として別設計にします。その場合も、名簿表示を最小限にし、端末管理、短時間セッション、自動ログアウトが必須です。

## 実装ステップ

### Step 1: 児童ログインUIを作る

- 生徒、先生、管理者のログインタブを分ける。
- 生徒タブではメールアドレスを出さず、児童番号とあいことばでログインする。
- ログイン前に児童名一覧を表示しない。

### Step 2: ログイン後の自動児童選択

- `lesson_user_access` が `student` 1件なら、その `user_data_id` を自動で `currentUser` にする。
- 児童は名前選択画面を通らず、すぐ練習メニューへ進む。

### Step 3: 児童アカウント作成補助

- 児童データからログインID一覧CSVを出す。
- Supabase Auth User IDと `student_...` の紐づけSQLを生成する。
- ログインカード印刷用CSVを出す。

### Step 4: セッション安全化

- 児童は一定時間操作がなければ自動ログアウトする。
- 共有端末ではログイン状態を残しにくくする。
- ログアウトボタンを常に分かりやすく置く。

### Step 5: 本番前確認

- 未ログインで名簿が見えない。
- 児童ログインで本人データだけ見える。
- 別児童の `user_data` が読めない。
- 先生ログインで許可グループだけ見える。
- 先生ログインでは進捗保存されない。
- 管理者ログインだけが復元、削除、設定変更できる。
